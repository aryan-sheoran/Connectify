## 1. The Three-Store Architecture — Overview

Connectify uses two databases plus a real-time layer, each chosen for a specific job:

| Store | Role | Why this store |
|---|---|---|
| **PostgreSQL** | Permanent structured data — users, rooms, memberships, refresh tokens, messages | Relational integrity, foreign keys, complex queries |
| **Redis** | Speed layer — caches, presence, token blocklist | Sub-millisecond reads, TTL expiry, pub/sub |
| **Socket.IO** | Real-time event bus over WebSockets | Bi-directional push, room broadcasting |

None of these overlap. PostgreSQL never stores ephemeral data. Redis never stores anything that must survive a restart. Socket.IO never persists anything — it only moves data between clients.

## 2. PostgreSQL — The Source of Truth

### Connection Setup

`config/db.js` creates a single **connection pool** (max 20 clients) using the `pg` library. A pool means the app keeps several open TCP connections to Postgres and reuses them across requests — instead of opening a new connection for every query (which is expensive). On startup the pool immediately tests one connection and logs success.

```
App boots
  → db.js imported → Pool created (up to 20 idle connections)
  → pool.connect() fires to test the connection
  → If OK: logs ✅ PostgreSQL connected
  → If fails: logs ❌ and the startup continues (pool will retry on first query)
```

### Schema and What Each Table Does

**`users`**
Stores one row per Google account. No password. `google_id` is the stable identifier from Google's OAuth. `username` is derived from the display name at first login with a 4-char random suffix to avoid collisions. `avatar_url` starts as the Google profile photo and can be updated by the user.

**`rooms`**
One row per chat room. `id` is a human-readable sequential string like `ROOM001` generated by `generateRoomId.js` — it queries the highest existing `ROOM%` ID and increments it. `created_by` is a FK to `users(id)` with `ON DELETE SET NULL`, so the room survives even if the creator deletes their account.

**`room_members`**
A pure join table — `(user_id, room_id)` composite primary key prevents duplicate memberships at the database level. `joined_at` records when the join happened (used for ordering in `getMyRooms`).

**`refresh_tokens`**
Every time a user logs in via Google, a new refresh token row is inserted. On logout, all of that user's tokens are marked `revoked = TRUE`. The `expires_at` column means expired tokens are automatically filtered out of validity queries. The `token_hash` column stores the raw refresh token JWT (the schema comment says SHA-256 hash but current code stores it raw — this is a known issue to fix).

**`messages`** *(updated architecture — PostgreSQL, not MongoDB)*
The updated `socket.js` inserts messages directly into PostgreSQL with a `messages` table. The schema needs a `messages` table (see the completion guide for the `ALTER` SQL). This makes the architecture simpler: one fewer database to manage.

### How PostgreSQL Queries Are Parameterized

Every query in the codebase uses `$1, $2, ...` placeholders and passes values as a separate array:

```js
pool.query('SELECT * FROM users WHERE id = $1', [userId])
```

The `pg` driver sends the query and values separately to Postgres. Postgres compiles the query plan once and substitutes the values safely — SQL injection is structurally impossible because user input never touches the query string.

### Indexes and Why They Exist

```sql
idx_room_members_user   ON room_members(user_id)   -- fast "what rooms has this user joined?"
idx_room_members_room   ON room_members(room_id)   -- fast "who is in this room?" (used by Redis cache population)
idx_rooms_type          ON rooms(type)             -- fast filter of public rooms in getAllRooms
idx_refresh_tokens_user ON refresh_tokens(user_id) -- fast lookup on logout (revoke all tokens)
idx_users_google_id     ON users(google_id)        -- fast upsert on every OAuth login
```

Without `idx_room_members_user`, the `getMyRooms` query would do a full table scan. With thousands of members, this matters.

## 3. Redis — The Speed Layer

### Connection Setup

`config/redis.js` creates a single Redis client using the `redis` v4 library and calls `connect()` immediately. The client emits `connect` and `error` events that are logged. The same client instance is exported and shared across the entire app — there is no pooling needed because Redis uses a single persistent TCP connection and handles pipelining internally.

### What Redis Stores and Why

Redis is an in-memory key-value store. Everything in it has a TTL (time to live) — if the server restarts or the TTL expires, the data is gone. That is intentional — Redis only stores things that can be safely reconstructed.

#### Key 1 — Room Membership Cache

```
Key:   room:{ROOMID}:members        e.g. room:ROOM001:members
Type:  Redis SET (unordered collection of unique strings)
TTL:   300 seconds (5 minutes)
Value: Set of user UUID strings    e.g. {"uuid-a", "uuid-b", "uuid-c"}
```

**Why it exists:** Every time a Socket.IO message is sent or a message history is loaded, the server must verify the user is a member of the room. Without this cache, every single event would require a PostgreSQL query. With the cache, that check is a Redis `SISMEMBER` call — a few microseconds instead of a few milliseconds.

**How it is populated:** On the first cache miss, the server queries `room_members` in Postgres for all members of that room, adds them all to the Redis SET with `SADD`, and sets the TTL to 5 minutes. Every subsequent check within those 5 minutes hits only Redis.

**How it stays fresh:** The TTL is 5 minutes. If someone joins or leaves a room, the PostgreSQL table is updated immediately. The Redis cache will be slightly stale for up to 5 minutes — acceptable because the join/leave REST endpoints are the source of truth, and the cache is only used for the membership check on message send (not for displaying the member list).

#### Key 2 — Online Presence

```
Key:   user:{userId}:online         e.g. user:uuid-a:online
Type:  String
TTL:   35 seconds
Value: The roomId the user is currently in  e.g. "ROOM001"
```

**Why it exists:** To know which users are "online" in a room at any given moment without polling PostgreSQL. The key expires automatically after 35 seconds. The Socket.IO `pingInterval` is 25 seconds — so as long as the client is connected and active, the socket server calls `setOnline()` which resets the TTL back to 35 seconds on every message sent or room join.

**How it works:** If a user disconnects cleanly (closes tab), the `disconnect` event fires and `redis.del()` removes the key immediately. If the connection drops silently (network failure), the `disconnect` event may not fire — but the 35-second TTL ensures the key expires automatically. Checking whether a user is online is as simple as checking if `user:{userId}:online` exists in Redis.

#### Key 3 — Token Blocklist

```
Key:   blocklist:{accessToken}      (the full JWT string)
Type:  String
TTL:   900 seconds (15 minutes — matches access token lifetime)
Value: "1"
```

**Why it exists:** JWT access tokens are stateless — once issued, they are valid until they expire. There is no way to invalidate them from the server side... unless you keep a blocklist. When a user logs out, their access token is added here. The `authenticate` middleware checks this key on every protected request. If the key exists, the request is rejected even though the JWT signature is technically valid.

**Why 15-minute TTL:** Access tokens expire in 15 minutes anyway. After that, the blocklisted token would be rejected by JWT verification before the Redis check even matters. So the TTL matches the token lifetime — after 15 minutes Redis cleans up the key automatically with no manual intervention needed.

#### Key 4 — Refresh Token Revocation (PostgreSQL, not Redis)

Refresh tokens are revoked in **PostgreSQL** (not Redis) because they live for 7 days and must survive Redis restarts. The `revoked` boolean column in `refresh_tokens` is the mechanism. On logout, `UPDATE refresh_tokens SET revoked = TRUE WHERE user_id = $1` fires.

### Redis Commands Used in This Project

| Command | Where Used | What It Does |
|---|---|---|
| `SISMEMBER key member` | `isMember()` in socket.js | Returns 1 if userId is in the room's member SET |
| `SADD key member [...]` | Cache population in socket.js / messageController | Adds member UUIDs to the room's SET |
| `EXPIRE key seconds` | After SADD | Sets the 5-minute TTL on the membership SET |
| `SETEX key seconds value` | `setOnline()`, token blocklist | Sets a key with TTL in one atomic command |
| `GET key` | Token blocklist check in authMiddleware | Returns the value or null |
| `DEL key` | Disconnect handler, leave_room | Removes a key immediately |

## 4. Socket.IO — The Real-Time Layer

### How Socket.IO Works Fundamentally

Socket.IO is not pure WebSockets — it is a layer on top that adds:
- Automatic fallback to HTTP long-polling if WebSockets are unavailable
- Reconnection logic
- Rooms (named channels for broadcasting)
- Acknowledgement callbacks (request-response over a socket)
- Heartbeats (ping/pong) to detect dropped connections

When a client calls `io('/')`, the browser first makes an HTTP request to negotiate the connection (`/socket.io/?EIO=4&transport=polling`). If WebSockets are available, it upgrades to a WS connection. The Vite dev proxy handles this upgrade: `ws: true` in `vite.config.js` passes the `Upgrade: websocket` header through to the backend.

### Authentication at the Socket Level

The Socket.IO auth middleware in `socket.js` runs before any event handlers. It does this:

```
1. Read raw Cookie header from socket.handshake.headers.cookie
2. Parse the cookie string manually into a key-value object
3. Extract the access_token value
4. Call verifyAccessToken() — same JWT verification as the REST middleware
5. If valid: attach socket.userId and socket.username to the socket object
6. If invalid: call next(new Error(...)) — connection is rejected before it opens
```

This means unauthenticated WebSocket connections are rejected at the handshake level, before any rooms or events are involved.

### Socket.IO Rooms vs Database Rooms

Socket.IO has its own concept of "rooms" that is separate from the chat rooms in the database. A Socket.IO room is just a named channel — a Set of socket IDs. When you call `socket.join('ROOM001')`, Socket.IO adds that socket's ID to the internal Set named `'ROOM001'`. When you call `io.to('ROOM001').emit(...)`, Socket.IO looks up that Set and sends the event to every socket in it.

This is purely in-memory and resets on server restart. It is separate from the `room_members` PostgreSQL table. The database tracks *who is a member* (persistent). Socket.IO tracks *which sockets are currently connected to a room* (ephemeral).

### The join_room Event — Step by Step

```
Client emits:  join_room  { roomId: 'ROOM001' }

Server:
  1. Normalise: roomKey = 'ROOM001' (toUpperCase)
  2. isMember(socket.userId, roomKey):
       a. redis.sIsMember('room:ROOM001:members', userId)
       b. If hit → return true immediately
       c. If miss → query PostgreSQL room_members
                  → SADD all member IDs into Redis
                  → EXPIRE 300
                  → return whether userId is in the set
  3. If not a member → callback({ success: false })  [connection denied]
  4. socket.join('ROOM001')   [add this socket to Socket.IO's ROOM001 channel]
  5. redis.setEx('user:{userId}:online', 35, 'ROOM001')
  6. socket.to('ROOM001').emit('user_joined', { userId, username, roomId })
     [broadcast to all OTHER sockets in ROOM001 — not the joiner]
  7. callback({ success: true })

Client receives: callback with { success: true }
Other clients in ROOM001 receive: 'user_joined' event
```

### The send_message Event — Step by Step

```
Client emits:  send_message  { roomId: 'ROOM001', content: 'Hello!' }

Server:
  1. Validate: roomId and content must be present
  2. Sanitize: content.trim().slice(0, 2000)  [hard cap at 2000 chars]
  3. isMember() — same two-tier Redis/PostgreSQL check as join_room
  4. If socket.avatarUrl is not cached:
       → query PostgreSQL: SELECT avatar_url FROM users WHERE id = userId
       → cache it on socket.avatarUrl for future messages (avoids repeated DB queries)
  5. INSERT INTO messages (room_id, sender_id, sender_username, sender_avatar, content, sent_at)
       → returns the new message id
  6. redis.setEx('user:{userId}:online', 35, roomKey)  [refresh presence TTL]
  7. Build payload: { id, roomId, senderId, senderUsername, senderAvatar, content, reactions:[], sentAt }
  8. io.to('ROOM001').emit('new_message', payload)
     [broadcast to ALL sockets in ROOM001 — including the sender]
  9. callback({ success: true, message: payload })

All clients in ROOM001 (including sender) receive: 'new_message' event with the full message object
```

Note: The sender receives the message back from the server (not from their own emit). This ensures:
- The message they see has the server-assigned ID and server timestamp
- If the message fails to save, they do not see it
- All clients including the sender are perfectly in sync

### The disconnect Event

```
Client disconnects (tab closed, network drop, etc.)

Server:
  socket.on('disconnect', async (reason) => {
    redis.del('user:{userId}:online')   [remove presence key immediately]
  })
```

On clean disconnect this fires immediately. On silent network drop, the 35-second TTL on the presence key handles cleanup automatically — the `disconnect` event fires after `pingTimeout` (60 seconds) when the client stops responding to pings.

### Typing Indicators — Pure Pub/Sub

```
Client A emits:  typing_start  { roomId: 'ROOM001' }

Server:
  socket.to('ROOM001').emit('typing', { username: 'Alice', roomId: 'ROOM001' })
  [NO database writes — this is fire-and-forget]

Other clients in ROOM001 receive: 'typing' event → show "Alice is typing..."

Client A emits:  typing_stop  { roomId: 'ROOM001' }
Other clients receive: 'typing_stopped' → hide the indicator
```

No Redis, no PostgreSQL. Just Socket.IO routing the event between clients.

## 5. How They Work Together — Full Request Flows

### Flow 1 — User Logs In

```
Browser → GET http://localhost:5000/api/v1/auth/google
  → Passport redirects to Google's consent page

Google → GET /api/v1/auth/google/callback?code=...
  → Passport exchanges code for Google profile
  → passport.js: UPSERT INTO users (google_id, username, email, avatar_url)
  → Returns user object to googleCallback controller

googleCallback:
  → signAccessToken({ id, username })   → JWT valid 15 min
  → signRefreshToken({ id })            → JWT valid 7 days
  → INSERT INTO refresh_tokens (user_id, token_hash, expires_at)
  → res.cookie('access_token', ..., httpOnly)
  → res.cookie('refresh_token', ..., httpOnly)
  → res.redirect('http://localhost:5173/user-home')

Browser loads /user-home:
  → AuthContext.checkAuth() → GET /api/v1/auth/me
  → authMiddleware:
      a. Read access_token from cookie
      b. redis.get('blocklist:{token}') → null (not blocked)
      c. verifyAccessToken() → decoded { id, username }
      d. req.user = { id, username }
  → getAuthUser: SELECT from users WHERE id = req.user.id
  → Returns user object → AuthContext sets currentUser
```

### Flow 2 — User Opens a Chat Room

```
Browser navigates to /chat-room/ROOM001
  ChatRoomPage.fetchRoomAndConnect():

  Step 1 — Join via REST:
    POST /api/v1/rooms/ROOM001/join
    → joinRoom controller:
        a. SELECT room from rooms WHERE id = 'ROOM001'
        b. Check type !== 'private'
        c. Check user not already in room_members
        d. Check room not full
        e. INSERT INTO room_members (user_id, room_id)
    → 200 OK  (or 409 if already a member — frontend ignores 409)

  Step 2 — Fetch room details:
    GET /api/v1/rooms/ROOM001
    → getRoomById: SELECT r.*, COUNT(members), creator username
    → Returns room object

  Step 3 — Fetch message history:
    GET /api/v1/rooms/ROOM001/messages?limit=50
    → getRoomMessages:
        a. redis.sIsMember('room:ROOM001:members', userId)  → cache check
        b. If miss → SELECT from room_members, SADD to Redis, EXPIRE 300
        c. SELECT from messages WHERE room_id = 'ROOM001' ORDER BY sent_at DESC LIMIT 51
        d. Reverse slice to chronological order
    → Returns formatted messages array

  Step 4 — Connect Socket.IO:
    io('/', { withCredentials: true })
    → WS handshake with access_token cookie
    → socket.js auth middleware validates token → socket.userId set

    socket.emit('join_room', { roomId: 'ROOM001' })
    → isMember() → Redis hit (just populated in Step 3)
    → socket.join('ROOM001')
    → setOnline(userId, 'ROOM001')
    → callback({ success: true })
```

### Flow 3 — Token Refresh (Silent)

```
Access token expires (15 min after login)

User makes any authenticated request:
  GET /api/v1/users/me/rooms
  → authMiddleware: verifyAccessToken throws TokenExpiredError
  → Response: 401 { code: 'TOKEN_EXPIRED' }

Axios interceptor in api.js catches 401 + TOKEN_EXPIRED:
  → POST /api/v1/auth/refresh (withCredentials: true)
  → refresh controller:
      a. Read refresh_token from cookie
      b. verifyRefreshToken() → decoded { id }
      c. SELECT from refresh_tokens WHERE user_id AND token_hash AND revoked=FALSE AND expires_at > NOW()
      d. SELECT username from users WHERE id = decoded.id
      e. signAccessToken({ id, username })
      f. res.cookie('access_token', newToken, httpOnly)
  → 200 OK

Axios retries the original request (now with new cookie):
  GET /api/v1/users/me/rooms → succeeds
```

## 6. Data Lifecycle Diagrams

### Where Each Type of Data Lives

```
┌─────────────────────────────────────────────────────────────┐
│                        PostgreSQL                           │
│                                                             │
│  users            rooms           room_members              │
│  refresh_tokens   messages                                  │
│                                                             │
│  → Permanent. Survives restarts. The source of truth.       │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                          Redis                              │
│                                                             │
│  room:{id}:members   SET, TTL 5 min   Membership cache      │
│  user:{id}:online    String, TTL 35s  Presence tracking     │
│  blocklist:{token}   String, TTL 15m  Logged-out tokens     │
│                                                             │
│  → Ephemeral. Safe to lose. Rebuilt from PostgreSQL.        │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                        Socket.IO                            │
│                                                             │
│  Room channels      In-memory Sets of socket IDs            │
│  socket.userId      Auth identity attached to connection    │
│  socket.avatarUrl   Avatar cached per connection            │
│                                                             │
│  → Transient. Lost on disconnect. Rebuilt on reconnect.     │
└─────────────────────────────────────────────────────────────┘
```

### Membership Check Decision Tree (Used on Every Message)

```
send_message received
        │
        ▼
redis.sIsMember('room:{roomId}:members', userId)
        │
   ┌────┴─────┐
  HIT        MISS
   │           │
   │           ▼
   │    SELECT user_id FROM room_members WHERE room_id = ?
   │           │
   │    ┌──────┴───────┐
   │  ROWS          NO ROWS
   │    │               │
   │    ▼               ▼
   │  SADD all      return false
   │  member IDs    (not a member)
   │  EXPIRE 300
   │    │
   │    ▼
   │  Is userId in that set?
   │    │
   │  ┌─┴──┐
   │ YES   NO
   │  │     │
   ▼  ▼     ▼
ALLOWED  DENIED
```
